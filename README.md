1. What you’re seeing isn’t a crash or a true “hang” — it’s a **fully blocking initialization path encountering total hardware absence**. In your test, nothing is connected to the Pico, so every hardware-facing operation silently fails. Your `init()` sequence is written under the assumption that hardware exists but may be temporarily unresponsive, so it performs **long waits, retries, and recovery steps**. The raw UART test sends `"AT"` but receives nothing because the RX line is floating; then the code deliberately sleeps, drains buffers, toggles pins, and finally runs `send_at_retry()` which itself waits for timeouts and retries multiple times. On top of that, your sensor manager is also doing **multi-attempt I2C initialization with recovery cycles**. When you stack all of this together, the system spends **many seconds inside blocking delays**, giving the impression that it has frozen. In reality, it’s just executing exactly what it was programmed to do: *wait patiently for hardware that does not exist*. The key issue is that your code does not distinguish between **“hardware temporarily misbehaving”** and **“hardware completely absent”**, so it wastes time trying to recover something that can never respond.

What you should do is redesign the initialization to be **failure-aware and time-bounded**. First, introduce a **fast-fail mechanism**: attempt a quick probe (e.g., a single `AT` with a very short timeout like 100–200 ms). If there is zero RX activity, immediately classify it as “no device present” and skip retries. Second, impose a **global initialization timeout** (for example, 2–3 seconds) so that no matter how many subsystems fail, the system regains control quickly. Third, reduce or eliminate long blocking delays inside init — especially multi-second sleeps and repeated retries — or make them conditional on actually detecting some response. Fourth, improve error classification by distinguishing cases like **no UART activity, timeout, or explicit modem error**, so higher-level logic can react intelligently. Finally, for a robust system (especially in embedded/flight contexts), move toward a **non-blocking state-machine-based initialization**, where each step progresses over time without halting the CPU. This way, even if every sensor and modem is missing, your system continues running, reports faults cleanly, and never “feels stuck.”
